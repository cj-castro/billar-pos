## Directory Purposes

**backend/app:**
- Purpose: Main Flask application package; contains all backend business logic
- Contains: Models (SQLAlchemy ORM), API routes (blueprints), services (stateless logic), WebSocket handlers
- Key files: `__init__.py` (app factory), `extensions.py` (extension initialization), `config.py` (environment config)

**backend/app/api:**
- Purpose: REST API endpoint definitions organized by domain/feature
- Contains: Flask blueprints for auth, resources, tickets, queue, inventory, menu, reports, users, waiting_list, cash_session
- Pattern: One blueprint per domain; routes prefixed with `/api/v1/{domain}`

**backend/app/models:**
- Purpose: SQLAlchemy ORM model definitions mapping to database tables
- Contains: User, Resource, PoolTableConfig, Ticket, TicketLineItem, LineItemModifier, LineItemPromotion, PoolTimerSession, MenuItem, MenuCategory, ModifierGroup, Modifier, MenuItemModifierGroup, InventoryItem, ModifierInventoryRule, StockMovement, Promotion, AuditLog, CashSession, Expense, TipDistributionConfig, WaitingListEntry
- Convention: Class name = singular, table name = plural (e.g., `User` → `users`)

**backend/app/schemas:**
- Purpose: Marshmallow schema validators for request body validation
- Contains: auth_schemas.py, menu_schemas.py, resource_schemas.py, ticket_schemas.py
- Usage: Applied in API routes to validate `request.get_json()` before processing

**backend/app/services:**
- Purpose: Stateless business logic functions used by multiple API routes
- Contains: 
  - `billing.py` - Pool table billing calculations (PER_MINUTE, ROUND_15, PER_HOUR modes)
  - `inventory_svc.py` - Stock tracking, consumption rules for menu items
  - `promotion_svc.py` - Promotion application and discount calculations
  - `audit_svc.py` - Centralized audit logging of state changes
- Convention: Functions import models and db as needed; no shared state

**backend/app/sockets:**
- Purpose: WebSocket event handlers for real-time client synchronization
- Contains: Connection handlers (connect/join/leave/disconnect), event definitions
- Events: `floor:update` (resources changed), `waiting:update` (waiting list changed), `ticket:update` (individual ticket changed)

**frontend/src:**
- Purpose: Root source directory for React application
- Contains: All React components, hooks, stores, API client, internationalization

**frontend/src/pages:**
- Purpose: Full-page route components (one per route defined in App.tsx)
- Contains: FloorMapPage, TicketPage, KitchenQueuePage, BarQueuePage, LoginPage, manager/* (ManagerDashboard, ReportsPage, InventoryPage, MenuManagementPage, UsersPage, PoolTableConfigPage, CashSessionPage, TableManagementPage, SettingsPage)
- Convention: Named with "Page" suffix; handle routing, data fetching, complex state

**frontend/src/components:**
- Purpose: Reusable UI components and modal dialogs
- Contains: ResourceCard (table/seat display), AddItemModal, TransferModal, ManagerPinDialog, NavBar, WaitingListPanel
- Convention: Named without "Page" suffix; accept props, emit callbacks; no routes

**frontend/src/stores:**
- Purpose: Global state management using Zustand
- Contains: 
  - `authStore.ts` - User identity, access token (persisted to localStorage)
  - `floorStore.ts` - Resources list and update methods (ephemeral)
- Convention: Exported as `useAuthStore()` and `useFloorStore()` hooks

**frontend/src/hooks:**
- Purpose: Custom React hooks encapsulating stateful logic
- Contains:
  - `useSocket.ts` - WebSocket context provider and hook
  - `useTimer.ts` - Elapsed time calculation for pool timers
  - `useEscKey.ts` - Keyboard listener for closing modals
  - `useLanguage.ts` - Language preference and i18next integration
- Convention: Exported as named hooks; wrap providers or return state/callbacks

**frontend/src/api:**
- Purpose: HTTP client and API configuration
- Contains: `client.ts` - Axios instance with JWT interceptor and error handling
- Configuration: Base URL `/api/v1`, Bearer token from authStore, auto-logout on 401

**frontend/src/i18n:**
- Purpose: Internationalization (i18next) translations
- Contains: `en.ts` (English), `es.ts` (Spanish), `index.ts` (initialization)
- Usage: Consumed via `useTranslation()` hook in components

**frontend/src/utils:**
- Purpose: Utility functions for specialized tasks
- Contains: `printReceipt.ts` (format and print ticket), `printCashReconciliation.ts`, `logoBase64.ts`

## Naming Conventions

**Files:**

- Backend: `snake_case.py` for modules; Blueprint files in `api/` match domain (auth.py, tickets.py, resources.py)
- Frontend: `PascalCase.tsx` for React components; `camelCase.ts` for hooks, utilities, stores
- Exceptions: Config files follow lowercase (config.py, extensions.py, wsgi.py)

**Directories:**

- Backend: Lowercase plural nouns (api, models, schemas, services, sockets)
- Frontend: Lowercase feature names (pages, components, stores, hooks, i18n, utils)

**Functions:**

- Backend: `snake_case` function names; route handlers named by operation (login, open_ticket, add_line_item, close_ticket, verify_pin)
- Frontend: `camelCase` for functions and hooks; component functions start with capital letter (FloorMapPage, ResourceCard, handleOpenNew)

**Variables:**

- Backend: `snake_case` for variables, parameters, database columns
- Frontend: `camelCase` for variables and state; constants in context (e.g., `barOpen`, `isManager`, `showAddItem`)

**Types:**

- Frontend: TypeScript interfaces/types in files where used; Props interfaces defined per component (e.g., `interface Props { resource: ResourceState; ... }`)
- Backend: No explicit type hints in models (dynamic SQLAlchemy columns); some type hints in services

**API Routes:**

- Pattern: `/api/v1/{domain}/{resource}/{action}`
- Examples:
  - `POST /api/v1/tickets` - open ticket
  - `GET /api/v1/tickets/{id}` - fetch ticket
  - `POST /api/v1/tickets/{id}/items` - add line item
  - `DELETE /api/v1/tickets/{id}/items/{item_id}` - void item
  - `PATCH /api/v1/tickets/{id}/discount` - apply discount
  - `POST /api/v1/tickets/{id}/close` - close and pay
  - `GET /api/v1/resources` - list all resources
  - `POST /api/v1/resources` - create resource
  - `PATCH /api/v1/resources/{id}` - update resource

## Where to Add New Code

**New API Endpoint:**

1. Create handler function in `backend/app/api/{domain}.py` (create new file if domain doesn't exist)
2. Add route decorator: `@{domain}_bp.route('/path', methods=['GET/POST/etc'])`
3. Add JWT protection: `@jwt_required()`
4. Implement logic using models from `backend/app/models/`
5. Call services as needed from `backend/app/services/`
6. Log audit trail via `audit_svc.log()`
7. Emit WebSocket events for real-time sync: `socketio.emit('floor:update', {}, room='floor')`

**New Feature Page:**

1. Create file `frontend/src/pages/{FeatureName}Page.tsx`
2. Import from routes: React Router (`useNavigate`, `useParams`), stores (`useAuthStore`, etc.), hooks, client
3. Use React Query for data fetching: `useQuery({ queryKey: [...], queryFn: () => client.get(...) })`
4. Manage local UI state with `useState` for modals, forms, etc.
5. Call API via `client.post/patch/delete` with error handling via `react-hot-toast`
6. Add route in `frontend/src/App.tsx` with proper role guards via `RequireAuth`

**New Reusable Component:**

1. Create file `frontend/src/components/{ComponentName}.tsx`
2. Define `interface Props` with expected inputs
3. Export default function with Props destructuring
4. Use Tailwind classes for styling
5. Import in page components and pass as children or props

**New Global State:**

1. Create file `frontend/src/stores/{storeName}.ts`
2. Use `create<T>()` from Zustand for typed state object
3. If persisting to localStorage: wrap with `persist()` middleware
4. Export as `use{StoreName}()` hook
5. Consume in components via hook

**New Custom Hook:**

1. Create file `frontend/src/hooks/use{HookName}.ts`
2. Return state, callbacks, or effects needed by multiple components
3. Use built-in React hooks and other custom hooks internally
4. Export as named function starting with "use"

**New Utility Function:**

1. Create or add to file `frontend/src/utils/{utilName}.ts`
2. Export pure functions (no side effects if possible)
3. Import and use in components/pages

**New Database Model:**

1. Create file `backend/app/models/{entity}.py`
2. Define SQLAlchemy model class with primary key, columns, relationships
3. Add `to_dict()` method for API serialization
4. Import in `backend/app/__init__.py` init-db command so table is created
5. Create associated API blueprint in `backend/app/api/{domain}.py`

**New Service/Business Logic:**

1. Create file `backend/app/services/{service_name}.py`
2. Define pure functions (no side effects on app state)
3. Import models and db as needed
4. Import in API routes that use this logic
5. Avoid creating service classes; use module-level functions
